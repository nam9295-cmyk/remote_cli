# veremote — 클로즈 베타 목표 문서

## 1. 프로젝트 한 줄 정의
veremote는 **현재 로컬 작업 폴더를 active workspace로 연결하고, 텔레그램에서 간단한 원격 코딩 명령을 보내 결과를 확인할 수 있는 개인형/소수 사용자용 원격 바이브코딩 CLI 도구**이다.

## 2. 이번 목표
이번 단계의 목표는 **공개 제품 출시**가 아니라,  
**소수 사용자(클로즈 베타)가 실제로 설치해서 써볼 수 있는 안정적인 내부 베타**를 만드는 것이다.

즉, veremote를 다음처럼 포지셔닝한다.

- 대상: 터미널 사용이 익숙한 소수 사용자
- 용도: 누워서 간단한 수정, 요약, 확인, 스크린샷 받기
- 핵심 가치: **내 로컬 컴퓨터를 텔레그램으로 가볍게 원격 제어**
- 비목표: 범용 대규모 에이전트 플랫폼, 다중 사용자 SaaS, 공개 서비스 수준의 협업 제품

## 3. 핵심 사용자 시나리오

### 시나리오 A — 로컬 연결
1. 사용자가 프로젝트 폴더로 이동한다.
2. `veremote`를 실행한다.
3. 현재 폴더가 active workspace로 연결된다.
4. 텔레그램으로 연결 안내 메시지가 간다.

### 시나리오 B — 원격 명령
1. 사용자가 텔레그램에서 `/where`로 현재 연결 폴더를 확인한다.
2. `/run <prompt>` 또는 `/edit <prompt>`를 입력한다.
3. 현재 active workspace 기준으로 작업이 실행된다.
4. 결과 요약, 변경 파일, 상태, 필요 시 이미지가 텔레그램으로 돌아온다.

### 시나리오 C — 로컬 복구/직접 작업
1. 텔레그램에서 작업이 느리거나 실패한다.
2. 사용자가 로컬 컴퓨터에서 `veremote`를 열어 직접 확인한다.
3. 필요하면 로컬에서 다시 명령하거나 로그를 확인한다.

## 4. 제품 범위 (클로즈 베타 기준)

### 반드시 되는 것
- 현재 폴더를 active workspace로 연결
- 텔레그램에서 `/where`, `/run`, `/edit`
- 작업 상태 확인 (`/job`, `/tail` 등)
- Gemini 기반 실제 실행
- Codex는 보조 엔진으로 선택 가능
- 작업 결과 텍스트 요약 전송
- 변경 파일 목록 전송
- 필요 시 이미지/스크린샷 전송
- 로컬 CLI로 직접 확인/복구 가능

### 이번 단계에서 굳이 안 하는 것
- 다중 사용자 권한 시스템
- 여러 사용자의 workspace/session 동시 관리
- 공개 SaaS 수준의 인증/결제/관리 UI
- Slack/Discord 등 다중 채널 확장
- 과한 GUI/TUI polish
- 복잡한 멀티 세션 제품화

## 5. 사용자 대상

### 적합한 사용자
- 터미널 사용이 익숙한 사람
- Node/웹 프로젝트를 만지는 사람
- 텔레그램을 자주 쓰는 사람
- “누워서 간단히 수정/요약/확인하고 싶다”는 니즈가 있는 사람

### 적합하지 않은 사용자
- GUI만 쓰고 싶은 사람
- 개발 환경 설정을 싫어하는 사람
- 협업 툴/팀 플랫폼을 기대하는 사람
- 대규모 자동화 플랫폼을 기대하는 사람

## 6. 현재 제품 성격
veremote는 현재 **개인용 알파를 넘어, 클로즈 베타 직전의 내부 도구** 수준으로 본다.

즉:
- 아이디어 검증은 끝남
- 핵심 흐름은 실제로 동작함
- 이제 필요한 것은 “더 많은 기능”보다 “안정성, 설치성, 결과 품질”

## 7. 클로즈 베타 전에 반드시 보강할 것

### P0 — 필수
1. **보안 경계 최소 보강**
   - 허용된 텔레그램 chat만 처리
   - active workspace 또는 allowlist 밖 경로 접근 금지
   - `run` / `edit` 역할 분리
   - Pencil 작업은 fallback 텍스트 성공 처리 금지

2. **상태/결과 명확화**
   - success / failed / partial / export_failed 구분
   - `/job <id>`와 `/tail <id>`로 진행 상황 확인 가능
   - edit 결과에 변경 파일 목록과 요약 포함

3. **daemon/workspace 안정화**
   - active workspace 연결/해제/조회 안정적
   - daemon 중복 실행 방지
   - 연결 상태 확인 가능

### P1 — 매우 중요
1. **설치/설정 흐름**
   - `veremote init`
   - `veremote doctor`
   - 전역 설치 검증 (`npm install -g`)
   - clean machine 설치 테스트

2. **문서화**
   - README 설치법
   - 첫 실행 흐름
   - 텔레그램 연결 방법
   - 엔진 선택 방법
   - 기본 명령 예시

### P2 — 있으면 좋은 것
1. Pencil 작업 실제 PNG export 강제
2. 수정 후 스크린샷 전송
3. 결과 메시지 polish
4. 입력창/TUI/로고 polish
5. Codex 지원 품질 개선

## 8. Pencil 작업 정책
Pencil 작업은 일반 코드 수정과 다르게 **최종 결과가 텍스트가 아니라 이미지**여야 한다.

따라서 Pencil 작업에 대해서는 아래 정책을 적용한다.

- 중간 진행 상태를 텔레그램으로 보낸다.
  - 작업 시작
  - 수정 중
  - export 중
  - 이미지 전송 중
- 마지막에는 **실제 export PNG가 반드시 있어야 한다**
- fallback 텍스트나 가짜 placeholder 이미지로 success 처리하지 않는다
- 이미지가 없으면 `partial` 또는 `export_failed` 로 처리한다

즉, Pencil 작업은 “요약 성공”이 아니라 “실제 결과 이미지 확보”를 완료 기준으로 삼는다.

## 9. 엔진 전략
클로즈 베타 기준 기본 엔진은 **Gemini** 로 둔다.

이유:
- 원격 텔레그램 사용에서는 복잡한 대형 코딩보다 짧고 명확한 작업이 많음
- Gemini가 이런 요약/경량 수정/구조 확인에 더 잘 맞을 가능성이 큼
- Codex는 보조 엔진으로 유지하되, 복잡한 작업은 로컬에서 직접 쓰는 방향이 현실적임

운영 원칙:
- 기본 엔진: Gemini
- 보조 엔진: Codex
- 복잡한 리팩토링/대형 수정은 로컬 직접 작업 권장

## 10. veremote의 핵심 차별점
veremote의 핵심은 범용 거대 에이전트 플랫폼이 아니라,  
**내 로컬 개발 환경을 텔레그램으로 가볍게 원격 제어하는 개인형 코딩 리모컨**이라는 점이다.

핵심 차별점:
- 현재 폴더 active workspace 중심
- 텔레그램 원격 명령
- 로컬 CLI가 항상 백업/복구 수단
- 실무형 결과 전달(요약 + 변경 파일 + 이미지)
- “누워서 간단히 코딩한다”는 실제 사용 감각에 맞춤

## 11. 클로즈 베타 성공 기준
아래 조건을 만족하면 클로즈 베타 시작 가능으로 본다.

1. 설치가 너무 어렵지 않다.
2. 현재 폴더 연결이 안정적이다.
3. 텔레그램에서 `/where`, `/run`, `/edit` 가 신뢰성 있게 동작한다.
4. 실패 시 원인을 사용자가 이해할 수 있다.
5. Pencil 작업은 실제 이미지 결과를 돌려준다.
6. 소수 사용자 3~5명이 자기 환경에서 써볼 수 있다.
7. “누워서 간단히 수정/확인하기 좋다”는 반응이 나온다.

## 12. 비전
veremote는 처음부터 OpenClaw 같은 범용 거대 시스템을 목표로 하지 않는다.

대신:
- 작고 빠르고
- 현재 폴더 기준으로 직관적이며
- 텔레그램으로 원격 제어 가능하고
- 개인 실무 흐름에 잘 맞는

**작지만 강한 원격 바이브코딩 도구**를 목표로 한다.

## 13. 코덱스 작업 원칙
앞으로 코덱스는 아래 원칙으로 작업한다.

1. 공개 제품보다 **클로즈 베타 품질**을 우선한다.
2. 새 기능을 무작정 늘리기보다 **안정성/설치성/결과 품질**을 우선한다.
3. UI/TUI 화려함보다 **원격 수정이 실제로 잘 되는지**를 우선한다.
4. 웹은 메인 흐름이 아니라 보조/legacy 취급한다.
5. core는 CLI + daemon + telegram + worker + db 이다.
6. 기능을 붙일 때마다 **직접 테스트 방법**을 사용자 입장에서 설명한다.
