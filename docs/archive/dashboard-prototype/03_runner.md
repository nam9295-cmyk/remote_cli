# Task 03 — CLI Runner & Log Streaming

## 목표
웹에서 생성한 작업을 실제 CLI 실행으로 연결하고, 상태와 로그를 갱신한다.

## 이번 단계에서 구현할 것
- 엔진 실행 설정 구조 정의
- child_process.spawn 기반 실행기 구현
- 작업 상태 업데이트
- 로그 파일 저장
- 작업 상세 페이지에서 로그 표시
- 실행 중 / 완료 / 실패 상태 반영

## 엔진 구조 예시
엔진 설정은 최소 아래 정보를 가진다.
- id
- name
- command
- argsTemplate
- enabled

초기 엔진 예시:
- gemini
- codex
- custom

## 실행 흐름
1. 작업 생성
2. 사용자가 실행 버튼 클릭
3. 서버가 job 상태를 running 으로 변경
4. CLI 실행
5. stdout/stderr 로그 저장
6. 종료 코드에 따라 success 또는 failed 처리
7. 결과 요약 / 에러 메시지 저장

## 로그 처리
- 로그는 파일로 저장
- 작업 상세 페이지에서 표시
- 너무 길 경우 최근 구간 우선 표시
- 나중에 실시간 스트리밍 확장 가능하도록 구조를 분리

## UI 요구사항
- 작업 상세에 실행 버튼
- 실행 중 배지 표시
- 로그 영역은 monospace 스타일
- 실패 시 에러 영역 표시

## 보안 주의사항
- 임의 쉘 명령 전체 실행 금지
- 미리 정의된 엔진 command만 사용
- 작업 디렉토리 제한 가능하도록 구조 분리
- 위험 명령 직접 입력 방식은 피하기

## 산출물
- 실제 CLI 실행 가능
- 상태 변화 반영
- 로그 저장 및 표시
- 최소한의 안전장치 포함

## 하지 않을 것
- 텔레그램 알림
- 외부 공개 접속
- 복잡한 실시간 웹소켓 구조

## 완료 기준
- 작업 상세에서 실행 버튼으로 실제 프로세스를 시작할 수 있어야 한다.
- 로그 파일이 저장되어야 한다.
- 성공/실패 상태가 UI에 반영되어야 한다.
- errorMessage가 저장되어 실패 원인을 볼 수 있어야 한다.

## 코덱스 실행 프롬프트
```text
project_plan.md와 tasks/03_runner.md를 읽고 CLI 실행부를 구현해줘.
child_process.spawn으로 엔진별 실행 구조를 만들고, 작업 상태를 queued/running/success/failed로 갱신해줘.
로그는 파일로 저장하고 작업 상세 페이지에서 볼 수 있게 해줘.
임의 쉘 명령 전체 실행은 막고, 엔진 설정 기반으로만 실행하게 해줘.
변경 파일 목록과 다음 단계 제안도 함께 정리해줘.
```
